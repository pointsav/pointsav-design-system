Climate zone tokens are performance specification data embedded in the [[bim-token-what-it-is|BIM Token]] system. Each token records the thermal and structural performance requirements that apply to a given IFC element type within a registered climate zone. The data lives in `tokens/bim/climate-zones.dtcg.json` in the BIM token vault — either `pointsav-design-system` or a customer vault such as `woodfine-design-bim` — and is read by `app-orchestration-bim` at startup.

## What a climate zone token is

A climate zone token is not a user-selectable value. It is reference data: the performance requirements that a regulation or standard imposes on a specific element type in a specific climate zone. Each token entry encodes one performance parameter (for example, maximum U-value) for one element type (for example, `IfcWall`) in one zone (for example, `ASHRAE-5C`).

The DTCG 1.0 schema for a climate zone entry:

```json
{
 "$type": "bim-climate-zone",
 "zone_id": "ASHRAE-5C",
 "ifc_class": "IfcWall",
 "parameter": "max_u_value",
 "required_value": 0.104,
 "unit": "Btu/h·ft²·°F",
 "source_standard": "ASHRAE 90.1-2022 Table 5.5-5",
 "$value": "climate-zone-spec"
}
```

## File location in the vault

```
<token-vault>/
└── tokens/
 └── bim/
 └── climate-zones.dtcg.json
```

Each row in the file covers one combination of zone, element type, and performance parameter. A project site with three element types in a single climate zone will match three rows; the token resolver binds all matched rows to the element during placement.

## Token resolution workflow

Site assignment is the entry point. The project site is linked to an `IfcSpatialZone` (PredefinedType: `USERDEFINED`, ObjectType: `CLIMATE_ZONE`) carrying the applicable zone identifier (for example, `ASHRAE-5C`).

When an element token is placed, the token resolver reads `climate-zones.dtcg.json` and finds all rows where `ifc_class` matches the placed element type and `zone_id` matches the project's assigned zone. The matched rows are bound to the element as required property values and appear in the element's Regulation tab as Climate Zone rows. The resolved requirements are projected into the IFC-SPF archive as `IfcConstraint` records linked to the element via `IfcRelAssociatesConstraint`.

## Validation

`ifctester` with an IDS 1.0 contract generated from the climate zone token data verifies that every element satisfies the requirements for its type:

```bash
# From the app-orchestration-bim project root
ifctester --ids climate-zones-bc-5c.ids model.ifc --report json
```

The IDS contract file for a climate zone is generated by the token vault tooling (planned for v0.0.3). At v0.0.1, the IDS contract must be authored manually from the `climate-zones.dtcg.json` parameter rows.

## Adding a new climate zone

To add performance data for a new zone:

1. Open `tokens/bim/climate-zones.dtcg.json` in the token vault.
2. Add rows for each element type and performance parameter in the new zone using the existing row structure.
3. Commit to the token vault repository: `feat(climate-zones): add <zone-id> performance parameters`.
4. Restart `app-orchestration-bim` — the vault is read at startup.
5. If the zone requires an IDS constraint file, author the IDS file and add it to `climate/<zone-id>/` in the vault.

| ASHRAE-7A | Very cold | IfcWall, IfcSlab |
| NBC-Zone-6 | Canadian cold climate | IfcWall, IfcSlab, IfcRoof |
